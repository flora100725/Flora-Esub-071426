# 🌐 TUDID x WHO MeDevIS 智慧雙軌決策 Agent 系統

## 1. 系統概述與願景 (System Overview & Vision)

本系統旨在打造一個部署於醫療機構、代理商或監管機構本地端（Local）的智慧型 Web 應用程式。透過結合台灣衛福部食藥署（TFDA）的 TUDID（台灣醫療器材單一識別碼資料庫）與世界衛生組織（WHO）於 2024 年推出的 **MeDevIS（醫療器械資訊系統）**，本系統利用大語言模型（LLM）驅動的 **Agentic 關鍵推理技術**，消弭台灣本土醫器許可證數據與國際標準命名體系（EMDN/GMDN）及臨床規範之間的巨大資訊不對稱。

系統採用 **Streamlit** 作為輕量化、高響應的前端與後端整合框架，確保醫療資訊工程師（Bioinformatics/Clinical Engineers）或法規專員（RA）能夠在隱私安全、數據不外流的本地環境下，一鍵上傳 TUDID 數據集（或使用內建預載範例），並由 AI Agent 自動與 WHO 遠端數據或本地鏡像進行動態語意對接、相容性審查、耗材推薦，最終一鍵輸出臨床招標與法規合規報告。

## 2. 核心功能規格 (Core Functional Specifications)

### 2.1 雙軌資料導入模組 (Dual-Track Data Ingestion)

* **自訂 TUDID 上傳：** 支援 `CSV`、`Excel (.xlsx)` 格式檔案上傳。系統需內建字串清洗機制（Data Cleaning），自動去除品名中常見的重複雙引號（如 `"""富江醫學"""`）及特殊全半形符號，並進行欄位正規化校驗。
* **內建預載數據（預設資料集）：** 系統內建預載你所提供的台灣經典醫器數據（包含牙科植體、人工水晶體、內視鏡取石籃等），點擊「載入範例數據」即可在無外接檔案情況下完整體驗所有 Wow 功能。
* **基礎 DI 與發碼機構解析：** 系統需能自動辨識 `GS1` 或 `HIBCC` 發碼機構，並根據基本 DI 欄位進行去重（De-duplication）與主鍵索引索引。

### 2.2 語意對接代理 (Semantic Mapping Agent) — *Wow Feature 1*

* **國際命名法多重對應：** 由於 TUDID 僅包含 TFDA 次類別碼（如 `M.3600`、`F.3640`），AI Agent 將利用 LLM 的醫學語意理解能力，提取「中文品名」、「英文製造商名稱」與「醫器次類別一」，動態與 WHO MeDevIS 的 `EMDN Code` 與 `GMDN Code` 進行多條件模糊匹配。
* **置信度評分系統（Confidence Scoring）：** Agent 對於對接結果需給出 $0\%$ 至 $100\%$ 的置信度評級，並將推理鏈（Chain of Thought, CoT）公開顯示於介面上（例如：「*因為中文品名包含『人工水晶體』且次類別為 M.3600，對應 WHO 核心器械 Analyser, ophthalmic...*」）。

### 2.3 醫院基礎設施相容性主動審查 (Infrastructure Dependency Checker) — *Wow Feature 2*

* **動態環境依賴提取：** 當用戶在 TUDID 中查到某高階雷射或手術系統時，系統會同步拉取 WHO MeDevIS 中定義的 `infrastructure_dependencies`。
* **落地安裝警告矩陣：** 主動揭露該醫器是否需要：
* **電力依賴：** 主電力穩定度、是否建議穩壓器（Voltage Stabilizer）、應急不斷電系統（UPS/UPS 備援）。
* **環境控制：** 空調需求、溫濕度精準限制（如高階雷射要求的 18-28°C、濕度減低 70%）。
* **資訊系統整合：** 是否支援院內 HIS、實驗室 LIS 或影像儲存傳輸系統（PACS）。
* **醫療廢棄物：** 是否涉及生物危害廢棄物處理（Biological Hazardous Disposal）。



### 2.4 智慧臨床耗材套裝推薦 (Automated MeDevPacks Suggester) — *Wow Feature 3*

* **臨床程序套件（MeDevPacks）聯動：** 當使用者選定某一 TUDID 核心設備（如骨內植體 `F.3640`），Agent 將自動將其歸類至 WHO 定義的臨床術式（如牙科手術），並自 WHO 資料庫中帶出該術式推薦的 `Set（套組）`、`Kit（工具包）`、`Module（模組）` 及 `Consumables（耗材）` 清單。
* **在地化替代品搜尋：** Agent 會反向在用戶上傳的 TUDID 資料庫中，搜尋與這些 WHO 推薦耗材相匹配的台灣現存有效許可證品項，實現「國際規範、在地採購」的自動化推薦。

### 2.5 許可證失效與供應鏈地緣風險預警 (Regulatory & Watchdog) — *Wow Feature 4*

* **許可證效期倒數計時：** 系統自動比對 TUDID 中的「有效日期」，針對 6 個月內即將到期的許可證顯示紅色警戒，並由 Agent 自動推薦同類別、且效期尚長的替代品。
* **地緣風險分析：** 提取「製造廠國別」（如 CN, DE, US, TR），結合外部公開的供應鏈穩定度指標，若某項核心醫器高度集中於單一國別，Agent 將彈出「供應鏈集中度風險提示」。

### 2.6 智慧政府採購招標文件生成器 (RFP Document Generator) — *Wow Feature 5*

* **技術規格說明書自動化：** 用戶可勾選 TUDID 中的一項或多項醫器，AI Agent 將結合台灣政府採購法格式與 WHO 官方的 Technical Specification，一鍵生成繁體中文的「醫療器材招標規格與技術需求說明書 (RFP)」。

## 4. 後端架構與 Agent 運作邏輯 (Backend Architecture & Agent Logic)

本本地端 App 雖然介面輕量，但後端設計了完整的 RAG（檢索增強生成）與 工具調用（Tool Calling）雙層架構。

### 4.1 資料庫與索引設計

1. **DataFrame 內存記憶體（In-Memory Storage）：** 對於用戶上傳的 TUDID 資料，後端使用 `pandas.DataFrame` 進行即時操作，並為「許可證字號」、「基本 DI」與「醫器次類別一」建立快速檢索索引。
2. **本地向量快取（Vector Cache / pgvector / FAISS）：** 系統在本地端利用輕量化向量工具（如 `FAISS`），將 TUDID 的「中文品名」與「醫器次類別中文描述」進行向量化（Embedding）。當 WHO MeDevIS 返回英文官方器械名稱時，不進行字面硬匹配，而是將英文品名翻譯後進行向量餘弦相似度（Cosine Similarity）計算，從而找出最匹配的 TUDID 本地紀錄。

### 4.2 Agent 工具箱（Agent Tools）與決策流程

AI Agent 核心擁有以下三個自訂工具：

1. `search_local_tudid(query_dict)`：負責在本地 Pandas 中進行多條件布林篩選。
2. `query_who_medevis_api(gmdn_or_emdn_code, keyword)`：負責向世衛組織 API 發起請求，獲取技術規格與基礎設施欄位。
3. `generate_rfp_template(tudid_data, medevis_data, hospital_context)`：負責整合兩端數據，組裝成符合台灣採購法規範的招標草案。

## 5. 數據安全性與隱私防護 (Data Security & Privacy)

由於醫療機構的採購數據、許可證庫存分布、廠商統一編號以及招標意向屬於高度敏感之內部機密資訊，本系統的本地端設計遵循以下隱私防護原則：

* **完全在地化處理（Zero-Data-Leak）：** 所有用戶上傳的 TUDID 檔案僅暫存於 Streamlit 後端的本地內存（RAM）中，當瀏覽器分頁關閉時自動銷毀，絕不上傳至任何第三方雲端空間。
* **大語言模型資安策略：** 支援切換為 **Local LLM 模式**（如透過 Ollama 連接本地端地植 Llama 3 醫療微調版模型），在此模式下，系統可達到 $100\%$ 物理隔離的斷網運作，完全杜絕資料外流風險。若使用雲端 API（如 OpenAI 或 Anthropic），系統將在發送請求前，自動對「申請商統一編號」等敏感數位進行去識別化（Anonymization）遮罩處理。
* **WHO API 憑證安全：** 用戶輸入的 WHO MeDevIS API Key 僅儲存於本地端的 `.env` 檔案或 Streamlit 執行的 Session State 中，不會被編碼或回傳。

## 6. 技術堆疊建議 (Recommended Tech Stack)

為了實現上述技術規格，本 Local Web App 建議採用以下 Python 庫進行建構：

* **Web 核心框架：** `streamlit (>= 1.35.0)` — 處理前端 UI 與雙向綁定。
* **數據處理核心：** `pandas (>= 2.1.0)`、`openpyxl` — 用於高效解析與清理 TUDID 數據。
* **AI Agent 與 RAG 框架：** `langchain` 或 `llama-index` — 用於建構 Agent 的思考鏈與工具調用。
* **本地向量計算：** `faiss-cpu` 與 `sentence-transformers` — 負責中英文醫器品名的向量語意對接。
* **高階互動圖表：** `plotly` — 用於繪製儀表板的地緣風險與過期預警圖表。
* **非同步 HTTP 客戶端：** `httpx` — 用於與 WHO MeDevIS API 進行高效、具備指數退避重試機制的異步通訊。

7. **發碼機構（GS1 vs HIBCC）特異功能：** 對於不同的發碼機構，其 UDI 條碼背後的解析逻辑不同（GS1 為 GTIN 規範，HIBCC 有其特定前綴）。系統是否需要支援直接在前端輸入「完整條碼字串」，由後端自動反向解析出基本 DI 並與 TUDID 比對？
